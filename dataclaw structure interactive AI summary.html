DataClaw
As-built architecture

A local-first data science agent platform with a plugin architecture and OpenClaw integration.

DataClaw is the realized version of the Titan vision. It is a two-part app: a Python backend with a plugin-based agent pipeline (FastAPI + LangGraph) and a React frontend, bridged to the OpenClaw agent runtime via channel and tool plugins. This page documents what was actually built.

01Core Architecture

DataClaw is a plugin-based agent platform with OpenClaw as its recommended runtime.

The system is split into two layers: DataClaw owns the product semantics (data access, notebooks, plans, projects, workspace governance) through a Python plugin system with swappable providers. OpenClaw provides the agent runtime (model routing, multi-provider support, memory, tool orchestration) and is bridged via two TypeScript plugins.

OpenClaw Agent Runtime
External agent runtime (recommended backend)
  • Multi-model routing across Anthropic, OpenAI, Google, and local models.
  • Gateway with WebSocket control plane and plugin system.
  • dataclaw-frontend channel plugin routes messages between UIs.
  • dataclaw-tools tool plugin exposes DataClaw's 34 tools to the agent.
  • Memory, skills, and identity managed by OpenClaw's own systems.
  • Bundled Docker deployment runs both in a single container.
DataClaw Product Layer
What DataClaw owns
  • Plugin system with entry-point discovery, dependency ordering, and provider injection.
  • Agent pipeline (LangGraph) with 9 hook points for plugin intervention.
  • Data plane with DuckDB, read-only safe SQL, dataset registry, profiling, and column analysis.
  • Notebook management with isolated Jupyter kernels per project.
  • Plan proposals with approval workflows and MLflow experiment tracking.
  • Project workspaces with scoped sessions, file management, and dataset access control.
  • React UI with chat, data browser, config, skills, and project management.
Design rule: DataClaw owns product semantics (connector policy, artifact identity, plan approval, workspace governance). OpenClaw owns agent capability (model routing, tool dispatch, session management). The provider layer abstracts LLM backends so routers and tools never call provider APIs directly.

08Key Takeaways

The shortest useful summary of DataClaw's architecture.

Plugin architecture, not a monolith.

7 backend plugins discovered via entry points, dependency-sorted, with DI. Each registers tools, hooks, routes, and optionally swaps providers. Adding new capabilities means writing a plugin, not modifying the core.

DataClaw owns product semantics. OpenClaw owns agent capability.

Connector policy, artifact identity, plan approval, workspace governance, and dataset access control are DataClaw's responsibility. Model routing, tool dispatch, memory, and session management are OpenClaw's. The provider layer abstracts the boundary.

Safe SQL by default, hooks everywhere.

Dataset queries are read-only DuckDB with single-statement validation and result caps. 9 hook points let plugins inject project context, filter dataset access, and intercept tool calls without touching the pipeline core.
